## 긍정적 발견 (이미 잘 구현된 부분)

1. **Prisma 파라미터 바인딩** — `$queryRaw`/`$executeRaw` 미사용, SQL Injection 차단
2. **입력 검증 프레임워크** (`lib/validation.ts`) — 타입/길이/범위/enum 체계적 검증
3. **파일 업로드 보안** (`app/api/uploads/route.ts`) — UUID 파일명, 10MB 제한, 확장자 화이트리스트, `path.basename()` + 경로 검증으로 path traversal 완전 차단
4. **세션 관리** (`lib/auth.ts`) — `crypto.randomUUID()`, 7일 만료, 비활성 사용자 즉시 무효화
5. **Rate Limiting** (`lib/rate-limit.ts`) — 로그인 5회 실패 → 15분 잠금
6. **감사 로깅** (`lib/audit-log.ts`) — 인증 이벤트, 데이터 CRUD 기록
7. **Self-protection** — 관리자 자기 자신 권한 해제/비활성화 불가
8. **CSV import** (`lib/csv-import.ts`) — 방어적 파싱 (날짜/숫자/불리언 각각 검증)
9. **에러 응답** — 내부 정보 미노출, 한국어 일반화 에러 메시지
10. **Non-root 컨테이너** — Dockerfile USER 지시문 적용

## 개발(Dev) 티켓

### SEC-DEV-001 [CRITICAL] POST /api/assets — ADMIN 역할 검증 누락

- **파일**: `app/api/assets/route.ts:102-105`
- **현상**: POST 핸들러에 `getCurrentUser()` 인증은 있으나 `user.role !== "ADMIN"` 체크 없음
- **위반**: guidelines.md — "모든 데이터 변경 API(POST/PUT/PATCH/DELETE)에는 역할 검증(RBAC) 필수"
- **영향**: 일반 USER 역할 사용자도 자산 등록 가능
- **수정 방법**:
  ```typescript
  // app/api/assets/route.ts — 105행 이후 추가
  if (user.role !== "ADMIN")
    return NextResponse.json({ error: "권한이 없습니다." }, { status: 403 });
  ```
- **관련 표준**: ISO 27001 A.9.1, ISMS-P 2.6

### SEC-DEV-002 [HIGH] 임시 비밀번호 생성에 Math.random() 사용

- **파일**: `app/api/admin/users/[id]/reset-password/route.ts:16`
- **현상**: `Math.floor(Math.random() * s.length)` — 암호학적으로 안전하지 않은 PRNG
- **영향**: 공격자가 PRNG 시드를 추론하면 임시 비밀번호 예측 가능
- **수정 방법**:
  ```typescript
  import { randomInt } from "crypto";
  const pick = (s: string) => s[randomInt(s.length)];
  ```
  Fisher-Yates shuffle의 `Math.random()`도 동일하게 교체:
  ```typescript
  const j = randomInt(i + 1);
  ```
- **관련 표준**: ISO 27001 A.10.1

### SEC-DEV-003 [HIGH] 세션 쿠키 secure 플래그 로직 불일치

- **파일**:
  - `app/api/auth/login/route.ts:105` → `secure: process.env.SECURE_COOKIE === "true"`
  - `app/api/auth/logout/route.ts:33` → `secure: process.env.NODE_ENV === "production"`
- **현상**: 로그인·로그아웃에서 서로 다른 조건으로 `secure` 플래그 결정
- **영향**: HTTPS 환경에서 로그아웃 시 쿠키 삭제 실패 가능 (secure 속성 불일치)
- **수정 방법**: 둘 다 `process.env.SECURE_COOKIE === "true"`로 통일
  ```typescript
  // app/api/auth/logout/route.ts:33
  secure: process.env.SECURE_COOKIE === "true",  // NODE_ENV 대신 SECURE_COOKIE 사용
  ```
- **관련 표준**: ISMS-P 2.5

### SEC-DEV-004 [MEDIUM] CRON 시크릿 비교 — timing-safe 미사용

- **파일**: `lib/cron-auth.ts:13`
- **현상**: `return token === secret` — 단순 `===` 비교
- **영향**: 이론적 타이밍 공격 가능 (폐쇄망 환경에서 실질적 위험은 낮지만 모범 사례 미준수)
- **수정 방법**:
  ```typescript
  import { timingSafeEqual } from "crypto";

  export function isCronAuthorized(request: NextRequest): boolean {
    const secret = process.env.CRON_SECRET;
    if (!secret) return false;
    const token =
      request.headers.get("x-cron-secret") ??
      request.headers.get("authorization")?.replace("Bearer ", "");
    if (!token) return false;
    const a = Buffer.from(token);
    const b = Buffer.from(secret);
    return a.length === b.length && timingSafeEqual(a, b);
  }
  ```
- **관련 표준**: ISO 27001 A.10.1

### SEC-DEV-005 [MEDIUM] 보안 헤더 미설정 — middleware.ts 부재

- **파일**: `middleware.ts` (신규 생성 필요, 프로젝트 루트)
- **현상**: X-Content-Type-Options, X-Frame-Options, Referrer-Policy 등 보안 헤더가 전혀 설정되지 않음
- **영향**: Clickjacking, MIME sniffing 공격에 노출
- **수정 방법**:
  ```typescript
  // middleware.ts (프로젝트 루트)
  import { NextResponse } from "next/server";

  export function middleware() {
    const res = NextResponse.next();
    res.headers.set("X-Content-Type-Options", "nosniff");
    res.headers.set("X-Frame-Options", "DENY");
    res.headers.set("Referrer-Policy", "strict-origin-when-cross-origin");
    res.headers.set("Permissions-Policy", "camera=(), microphone=(), geolocation=()");
    return res;
  }

  export const config = {
    matcher: ["/((?!_next/static|_next/image|favicon.ico).*)"],
  };
  ```
- **관련 표준**: ISO 27001 A.14.1, OWASP Secure Headers

### SEC-DEV-006 [LOW] 비밀번호 복잡성 요구사항 미적용

- **파일**: `app/api/admin/users/route.ts:47`, `app/api/auth/change-password/route.ts:17`
- **현상**: 8자 이상 길이만 검사. 대문자/소문자/숫자/특수문자 조합 요구 없음
- **영향**: 단순 비밀번호(예: `aaaaaaaa`) 허용
- **수정 방법**: 최소 영문+숫자 조합 검증 추가
  ```typescript
  const hasLetter = /[a-zA-Z]/.test(password);
  const hasDigit = /[0-9]/.test(password);
  if (!hasLetter || !hasDigit)
    return NextResponse.json({ error: "비밀번호는 영문과 숫자를 포함해야 합니다." }, { status: 400 });
  ```
- **관련 표준**: ISMS-P 2.5.3

## DevOps 티켓

### SEC-OPS-001 [HIGH] docker-compose.yml — 평문 크리덴셜 하드코딩

- **파일**: `docker-compose.yml:9-10, 35`
- **현상**: `POSTGRES_USER: asset_manager`, `POSTGRES_PASSWORD: asset_manager_pass`, `DATABASE_URL` 등에 평문 비밀번호 하드코딩
- **위반**: guidelines.md — "환경변수에 시크릿을 하드코딩하지 않는다"
- **수정 방법**: `.env` 파일에서 환경변수 치환 방식으로 변경
  ```yaml
  environment:
    POSTGRES_USER: ${POSTGRES_USER}
    POSTGRES_PASSWORD: ${POSTGRES_PASSWORD}
    POSTGRES_DB: ${POSTGRES_DB}
  ```
- **관련 표준**: ISO 27001 A.10.1, ISMS-P 2.7

### SEC-OPS-002 [HIGH] PostgreSQL 포트 외부 노출

- **파일**: `docker-compose.yml:12-13`
- **현상**: `"5432:5432"` — DB 포트가 호스트 외부에 노출
- **영향**: 네트워크상 누구나 DB에 직접 접속 가능 (폐쇄망이라 하더라도 내부 공격자 위험)
- **수정 방법**: localhost 바인딩 또는 포트 매핑 제거
  ```yaml
  ports:
    - "127.0.0.1:5432:5432"   # localhost만 접근 허용
  ```
  또는 app 컨테이너가 내부 네트워크로 통신하므로 `ports` 완전 제거도 가능
- **관련 표준**: ISO 27001 A.13.1, Zero Trust 마이크로 세그멘테이션

### SEC-OPS-003 [HIGH] SECURE_COOKIE 프로덕션 기본값 false

- **파일**: `docker-compose.yml:36`
- **현상**: `SECURE_COOKIE: "false"` — 프로덕션 환경에서도 쿠키가 HTTP로 전송
- **영향**: 중간자 공격(MITM)으로 세션 탈취 가능
- **판단**: 폐쇄망에서 HTTPS 미사용 시 `false` 유지가 불가피할 수 있음 → **HTTPS 도입 검토 포함**
- **관련 표준**: ISMS-P 2.5
