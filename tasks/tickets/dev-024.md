## 사용자 확정 (질문 응답)
- ① 임직원 부서 중복 노출 컬럼 하나로 통합
- ② 내용연수·감가상각 정책 적용
- ③ 목록 페이지네이션·정렬
- (안정성/접근성 번들은 이번 제외)

## 설계 결정
- 부서 중복: 상단 요약(이름/부서/이메일)과 조직 섹션(직급/회사/부서)에서 부서가 2회 노출.
  → 편집 가능한 조직 섹션을 단일 본거지로 두고 상단 요약에서 부서 제거. 조직 섹션 읽기뷰에
    레거시 free-text `department` 폴백 추가(orgUnit 미설정 직원 정보 보존).
- 내용연수/감가상각: 기존 LifecycleVisibilityProvider/useLifecycleVisible 패턴 재사용.
  useLifecycleVisible export → 하드웨어 page 의 감가상각 요약 블록·내용연수(usefulLife) 열을 게이팅.
  (게이지 셀은 이미 자가 게이팅됨)
- 하드웨어 정렬: API orderBy 는 Asset 최상위 필드만 → assetTag/manufacturer/assignee 등 관계
  필드는 서버 정렬 불가. 정렬은 클라이언트 유지, 페이지네이션은 "더 보기"로 다음 page 누적 로드.
- 임직원: 페이지는 서버 컴포넌트(prisma 직접). page(50/건) + 정렬을 searchParams 로 구현.
  `unassigned` 인메모리 필터 → DB where(`assignments: { none: { returnedDate: null } }`) 로 이동
  해야 페이지네이션 total 이 정확.

## 검증
- npm run build / npm run lint
- 비-SUPER_ADMIN + flag off 에서 감가상각 요약 비노출 수동 확인
