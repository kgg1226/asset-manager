## 배경 (사용자 요구)
ISMS-P 흐름도는 단계별(수집/이용·제공/파기)로 여러 장 — 페이지마다 독립된 연결 집합이 필요.
기존엔 visibleEdgeIds 가 all 뷰 전용이라 비-all 뷰가 전역 연결을 전부 노출("하나의 흐름이 모든 페이지에").

## 설계 — 화이트리스트의 "페이지 소속" 승격 (스키마 무변경)
- AssetLink 에 페이지 FK 를 박는 안(C-full)은 배제: 같은 흐름의 페이지 간 재사용 불가 + ISMS-P 흐름표
  export(전사 증적 합집합, dev-050)가 조각남. 소속(visibleEdgeIds) 방식은 연결 전역성을 보존.
- fetchGraph: 소속 필터를 전 뷰에 적용(뷰 linkType 필터와 교집합). null(레거시)=전체 표시 유지.
- 생성 3경로(새 페이지·예시 페이지·현재 뷰 저장) 모두 소속 초기화 — 새 페이지 [], 뷰 저장은 현재 캔버스 연결.
- 토글 일반화: 현재 뷰 조회분(allLinkIdsRef)을 소속에 유니온(표시)/차집합(숨김) — 타 유형 소속 불변.
  레거시 페이지는 all 뷰 전용(비-all 에서 배열 신설 시 타 유형이 전부 숨는 사고 방지).
- 저장측 가드 불변: flushSave 재작성은 all 뷰 전용(dev-049), 비-all 생성 편입은 dev-056 append-only.

## 알려진 한계 (문서화)
- all 뷰 자동저장은 캔버스 기준으로 소속을 재작성한다(기존 동작 유지). 비-all 뷰에서 새 자산을 배치하고
  연결하면 위치가 저장되지 않으므로(dev-049 가드, 기존 한계) all 뷰 재작성 시 해당 연결이 소속에서 빠질 수
  있다. 권장 워크플로: 페이지 구성(배치·연결)은 전체 뷰에서, 단계 뷰는 필터 열람용.
