## 배경
assetTag 는 "앱 레벨에서 중복 체크" 주석과 함께 사전 체크 3곳(create/update/bulk-tag) +
import dedup 만 있었고, dedup-asset-tag.mjs 스크립트("unique 제약 적용 전 정리")까지
준비돼 있었으나 정작 스키마 @unique 는 미적용 상태였다. check-then-write 는 동시 요청
레이스에 뚫리므로 DB 제약이 최종 방어선으로 필요.

## 구현
- HardwareDetail.assetTag → @unique (nullable 유지 — Postgres 는 NULL 다중 허용).
- 마이그레이션: CREATE UNIQUE INDEX "HardwareDetail_assetTag_key".
- 기존 앱 레벨 사전 체크는 유지(친절한 메시지·빠른 실패), DB 제약은 레이스 캐치.
  P2002 → handlePrismaError(uniqueMessage) 로 409 통일. bulk-tag 는 catch 자체가 없어 추가.
- 배포 안전: entrypoint 가 db push "전에" dedup-asset-tag.mjs 를 실행해 기존 중복을
  정리(최소 id 만 보존, 나머지 null) 후 제약 적용 — 기존 흐름 그대로 동작.
